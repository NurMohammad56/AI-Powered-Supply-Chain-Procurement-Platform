## গল্পের চরিত্র

পুরো গাইডে আমরা একটা উদাহরণ ব্যবহার করব — আপনার একটা গার্মেন্টস **"ঢাকা ডেনিম"**, আর
আপনার দরকার **১০০০ গজ কাঁচা তুলা (raw cotton)**। এই তুলা কেনা থেকে শুরু করে গুদামে তোলা,
ভবিষ্যৎ চাহিদা হিসাব করা — পুরো যাত্রাটাই এই সিস্টেম।

আর মাঝে মাঝে আমরা **৫টা গার্মেন্টসের** কথা বলব (ঢাকা ডেনিম, চট্টগ্রাম নিটওয়্যার, গাজীপুর
টেক্সটাইল, সাভার গার্মেন্টস, নারায়ণগঞ্জ অ্যাপারেল) — কারণ এই সিস্টেমে অনেক গার্মেন্টস একসাথে
চলতে পারে, কিন্তু প্রত্যেকের ডেটা সম্পূর্ণ আলাদা থাকে।

## ১. এই সিস্টেম আসলে কী? (এক বাক্যে)

**এটা একটা সফটওয়্যার যেটা একটা ফ্যাক্টরিকে তাদের কেনাকাটা আর স্টক (মাল) ম্যানেজ করতে সাহায্য
করে — আর AI দিয়ে আগে থেকে বলে দেয় সামনে কী লাগবে, যাতে মাল কখনো শেষ না হয় আর বাড়তি কিনে
টাকা নষ্ট না হয়।**

তিনটা স্তম্ভ:
- **Supply Chain** — ফ্যাক্টরির মাল কোথায় কত আছে (inventory)
- **Procurement** — supplier থেকে মাল কেনা (quote → অর্ডার → ডেলিভারি)
- **AI-Powered** — মানুষের আন্দাজের বদলে AI চাহিদা হিসাব করে আর রিপোর্ট লিখে দেয়

## ২. যে সমস্যা এটা সমাধান করে

আগে একটা ফ্যাক্টরি চলত **স্প্রেডশিট, ইমেইল, হোয়াটসঅ্যাপ আর আন্দাজের** ওপর। সমস্যাগুলো:

| পুরোনো সমস্যা | সিস্টেম যা দেয় |
|---|---|
| স্টকের হিসাব কেউ বিশ্বাস করে না | একটাই নির্ভরযোগ্য, লাইভ স্টক খাতা |
| হঠাৎ মাল শেষ হয়ে প্রোডাকশন বন্ধ | AI চাহিদা হিসাব + কম-স্টক সতর্কতা |
| বেশি কিনে টাকা আটকে যায় | কতটুকু কিনতে হবে AI বলে দেয় |
| supplier থেকে কেনা এলোমেলো | গোছানো quote → তুলনা → অর্ডার → রিসিট |
| রিপোর্ট বানাতে ঘণ্টার পর ঘণ্টা | সপ্তাহে একটা AI-লেখা রিপোর্ট অটো ইমেইল |
| কে কী করল কোনো রেকর্ড নাই | প্রতিটা গুরুত্বপূর্ণ কাজের অডিট ট্রেইল |

## ৫.১ — Auth মডিউল (দারোয়ান + পরিচয়পত্র 🚪🪪)

### এটা কী?
সিস্টেমের **সদর দরজা আর নিরাপত্তারক্ষী**। দুইটা প্রশ্নের উত্তর দেয়:
1. **তুমি কে?** (login)
2. **তুমি কোন factory-র, আর কী করার অনুমতি আছে?** (tenant + role)

> এখানেই tenant আলাদা রাখার পুরো ভিত্তি — কারণ **JWT টোকেনটা এই auth মডিউলই বানায়।**

### সবচেয়ে গুরুত্বপূর্ণ মডেল — `Factory`
- **`Factory` = একটা গার্মেন্টস = একটা tenant।** ৫টা গার্মেন্টস = ৫টা Factory ডকুমেন্ট।
- এটাই **একমাত্র মডেল যেটা tenant-scoped না** — কারণ Factory **নিজেই** tenant।
- `User` (কর্মী) আর `Session` (লগইন সেশন) — Factory-র অধীনে।

### Tenant কীভাবে তৈরি ও আটকে যায়

```
১. Register (নতুন গার্মেন্টস সাইন আপ)
   POST /auth/register
   → একটা Factory তৈরি (ধরুন _id = aaa...001)
   → একটা Owner User তৈরি, যার tenantId = aaa...001
        │
        ▼
২. Login → POST /auth/login
   → JWT টোকেন বানায়, ভেতরে সিল মারা: { tenantId: aaa...001, role: 'owner' }
   → টোকেন cryptographically signed (সরকারি সিলের মতো, জাল করা যায় না)
        │
        ▼
৩. প্রতি request-এ এই টোকেন যায়
   → সিস্টেম দেখে "ও আচ্ছা, ঢাকা ডেনিম-এর (aaa...001)"
   → এই tenantId দিয়েই সব ডেটা ফিল্টার হয়
```

### Role — একই factory-র ভেতরে কে কী পারবে
- **Owner** — সব পারে (টাকা দেখা, বড় সিদ্ধান্ত)
- **Manager** — দৈনন্দিন কাজ, PO অনুমোদন
- **Warehouse staff** — শুধু মাল নাড়াচাড়া

`role` বদলানো = **শুধু Owner**। সুরক্ষা: **শেষ Owner-কে মোছা যায় না**, নিজে নিজের role বদলানো যায় না।

### নিরাপত্তা (সহজ ভাষায়)
- **bcrypt cost ≥12** — পাসওয়ার্ড চুরি হলেও পড়া যায় না
- **JWT: 15 মিনিট access / 7 দিন refresh** — ছোট চাবি ঘন ঘন বদলায়
- **Refresh-token family + reuse detection** — পুরোনো চাবি আবার ব্যবহার হলে "চুরি!" বুঝে পুরো সেশন বন্ধ
- **JWT denylist (Redis)** — "logout everywhere" চাপলে সব চাবি তাৎক্ষণিক বাতিল
- **CSRF header + rate limit (5/15min)** — আক্রমণ ঠেকায়

## ৫.২ — Inventory মডিউল (গুদামের সত্য তথ্য 📦)

### এটা কী?
**"কোথায় কী মাল, কত আছে, কীভাবে নড়ছে"** — এর হিসাবের খাতা। পুরো সিস্টেমের **সত্যের উৎস**।
AI এখান থেকে পড়ে forecast করে, PO এখানে মাল যোগ করে।

### মডেলগুলো
- **`Warehouse`** — গুদাম
- **`ItemCategory`** — মালের ভাগ (তুলা, সুতা, জিপার...)
- **`Item`** — আসল মাল। ৪ ধরন: raw_material, finished_good, packaging, consumable
- **`StockBalance`** — এখন কত আছে (হিসাব করা)
- **`StockMovement`** — প্রতিটা নড়াচড়ার রেকর্ড (ঢুকল/বের হলো)

### সবচেয়ে গুরুত্বপূর্ণ নিয়ম — "append-only ledger" 📒

ব্যাংকের পাসবইয়ের মতো:

```
StockMovement = পাসবই (শুধু লেখা যোগ হয়, কখনো মোছা যায় না)
   +১০০০ গজ তুলা ঢুকল   (১লা তারিখ)
   -২০০ গজ ব্যবহার হলো   (৩রা তারিখ)
   -১৫০ গজ ব্যবহার হলো   (৫ই তারিখ)
        │
        ▼ (যোগ-বিয়োগ করে)
StockBalance = এখন আছে ৬৫০ গজ  (এটা হিসাব করে বের করা, "derived")
```

**কেন?** ইতিহাস কখনো মোছা যাবে না — "৩রা তারিখে কে ২০০ গজ নিল?" উত্তর সবসময় থাকা চাই।

> **`balance_audit` cron:** সপ্তাহে একবার মিলিয়ে দেখে পাসবইয়ের যোগফল আর current balance মিলছে
> কিনা; গরমিল থাকলে ঠিক করে।

### দুইটা গুরুত্বপূর্ণ হুক (এখানে PO-র সাথে জোড়া লাগে 🔗)
- **`incrementBalance`** — মাল ঢুকলে নিরাপদে (atomic, race ছাড়া) balance বাড়ায়
- **`clearLowStockIfResolved`** — নতুন মাল এলে "কম-স্টক" সতর্কতা মুছে দেয়

## ৫.৩ — Supplier মডিউল (কার কাছ থেকে কিনব? 🤝)

### Supplier মানে কী?
**বিক্রেতা/সরবরাহকারী।** প্রত্যেকের প্রোফাইলে থাকে: Contacts (লোকজন), Documents (কাগজপত্র),
Performance score (আগে কত ভালো কাজ করেছে)। যোগ/এডিট/তুলনা করা যায়।

### RFQ — দাম জানতে চাওয়া (সবচেয়ে গুরুত্বপূর্ণ)

**RFQ = Request For Quotation = "দাম জানাও"** অনুরোধ।

```
১. RFQ বানান (QuotationRequest)
   → কোন আইটেম, কত পরিমাণ, কোন supplier-দের দাওয়াত
        │
        ▼
২. প্রত্যেক supplier একটা গোপন "token" (চাবি) পায় (ইমেইলে)
        │
        ▼
৩. Supplier login ছাড়াই দাম জমা দেয়
   POST /public/quotations/responses/:token
   → token প্রমাণ করে "এই দাওয়াত আমারই"
```

**token কেন?** supplier তো আপনার সিস্টেমের ইউজার না, password নাই। তাই একবার-ব্যবহারের গোপন
চাবি দেওয়া হয়, যেটা দিয়ে সে শুধু **তার নিজের** দাম জমা দিতে পারে।

### Compare — কে সেরা অফার দিল?
1. **সংখ্যার হিসাব** — মেশিন নিখুঁতভাবে: কার দাম কম, কার lead time কম (১০০% অংক)
2. **AI-এর কথা** — সহজ ভাষায়: "Supplier B সস্তা, কিন্তু ২ দিন বেশি লাগবে"

> ⚠️ AI **নিজে supplier বাছে না**। সংখ্যা মেশিন দেয়, AI ব্যাখ্যা করে, **সিদ্ধান্ত মানুষ নেয়।**

### Accept — এখানেই Supplier আর PO জোড়া লাগে! 🔗
একজন supplier-কে **Accept** করলে সিস্টেম **অটো একটা Purchase Order (PO) বানায়** — ওই supplier-এর
দাম দিয়েই। **RFQ শেষ → PO শুরু।**

## ৫.৪ — PO মডিউল (আসল কেনাকাটা 📝)

### PO মানে কী?
**Purchase Order = ক্রয় আদেশ।** অফিসিয়াল অর্ডার ফরম। ধাপে ধাপে এগোয় একটা **state machine**
(অবস্থার সিঁড়ি) মেনে।

### State Machine — PO-এর জীবনযাত্রা

```
draft (খসড়া)
  │ submit
  ▼
pending_approval (অনুমোদনের অপেক্ষায়)
  │ approve (বস "হ্যাঁ")        │ reject (বস "না")
  ▼                             ▼
approved (অনুমোদিত)          rejected → এডিট করে আবার draft
  │ dispatch
  ▼
sent (supplier-এর কাছে গেছে)
  │ receive (কিছু মাল এলো)
  ▼
partially_received
  │ receive (বাকিটা এলো)
  ▼
fully_received
  │ close
  ▼
closed (শেষ)

(যেকোনো ধাপ থেকে cancel → cancelled)
```

**কেন এত কড়াকড়ি?** যাতে কেউ অনুমোদন ছাড়া `draft` PO সরাসরি `sent` বানিয়ে মাল চাইতে না পারে।

### `PO_STATE_RACE` — দুইজন একসাথে চাপলে?
দুইজন ম্যানেজার একই সময়ে "Approve" চাপলে সিস্টেম **CAS** কৌশলে একজনকে জিতিয়ে দেয়, অন্যজন পায়
`PO_STATE_RACE` ("রিফ্রেশ করুন")। ডেটা কখনো গোলমাল হয় না।

### Dispatch — supplier-কে অর্ডার পাঠানো
`approve` হলে `dispatch` করলে: **PDF বানায়** (PDFKit) → **R2-তে আপলোড** → লিংক সেভ → state `sent`
→ **supplier-কে ইমেইল** PDF-সহ।

### Receive — মাল আসা (এখানে সব মডিউল একসাথে! 🔗🔗)

```
মাল এলো (Receive)
   │
   ├─① Inventory-তে stock বাড়ে (StockMovement + StockBalance)
   ├─② Low-stock alert বন্ধ হয় (যদি stock যথেষ্ট হয়)
   ├─③ AI-কে নতুন forecast করতে বলে
   └─④ পুরো মাল এলে → "অর্ডার সম্পূর্ণ" ইমেইল
```

**"Best-effort, never roll back" মানে:** মাল রেকর্ড হলো কিন্তু ইমেইল ফেল করল — তবু **মাল আসার রেকর্ড
মুছবে না**। মূল কাজ নিরাপদ, পাশের ছোট কাজ আলাদাভাবে চেষ্টা করে।

**③-তে "6h staleness check" কেন?** ২০০ আইটেমের PO রিসিভ করলে ২০০টা forecast job হবে → AI খরচ বাড়বে।
তাই নিয়ম: "গত ৬ ঘণ্টায় forecast হয়ে থাকলে আবার করো না।"

## ৫.৫ — AI মডিউল (সিস্টেমের মস্তিষ্ক 🧠)

### এটা কী করে?
১. **Forecast** — "আগামী ৩০/৬০/৯০ দিনে কত তুলা লাগবে?"
২. **সাপ্তাহিক রিপোর্ট** — মালিককে সহজ ভাষার রিপোর্ট ইমেইল।

### Forecast পাইপলাইন (রান্নার রেসিপির মতো)

```
১. উপকরণ জোগাড় (prepareForecastContext)
   → গত ১৮০ দিনের ব্যবহারের হিসাব → প্যাটার্ন (গড়, ওঠানামা, বাড়ছে/কমছে, চক্র)
        │
        ▼
২. প্রশ্ন সাজানো (renderForecastPrompt)
        │
        ▼
৩. AI-কে জিজ্ঞেস: Groq (প্রধান) ──[ব্যর্থ হলে]──► Gemini (বিকল্প)
   → Circuit breaker: ৩ বার ফেল করলে ৬০ সেকেন্ড বিশ্রাম
        │
        ▼
৪. উত্তর পরিষ্কার (coerceForecast)
   → যাচাই → মেরামত → একদম না হলে সাধারণ অংকের হিসাব (baseline)
        │
        ▼
৫. সংরক্ষণ + Redis 24h cache + খরচের হিসাব + ড্যাশবোর্ডে live খবর
```

> **কখনো খালি হাতে ফেরে না:** Groq ফেল → Gemini → তাও না হলে অংক। **সবসময় একটা উত্তর পাবেন।**

### খরচ নিয়ন্ত্রণ 💰
- প্রতি factory-র **মাসিক সীমা (quota)**, subscription tier অনুযায়ী
- **`checkQuota` চলে AI কল করার আগেই** — সীমা শেষ হলে কল হবেই না
- ৮০% খরচে **সতর্কবার্তা**
- `provenance` = প্রতিটা forecast-এর হিসাবের খাতা (কোন AI, কত সময়, কত token)

### সাপ্তাহিক রিপোর্ট
৫ জায়গা থেকে তথ্য (stock + PO + supplier + balance + forecast) → AI লেখে → Markdown → HTML →
**PDF** → মালিকের ইমেইল।

## ৫.৬ — RPT মডিউল (রিপোর্ট/অ্যানালিটিক্স 📊)

শুধু **পড়ার (read-only)** রিপোর্ট, কোনো AI নাই — **খাঁটি অংক**।

- **Inventory turnover** — মাল কত দ্রুত ব্যবহার হচ্ছে?
- **Spend by supplier** — কোন supplier-এ কত টাকা গেল?
- **Supplier cost comparison** — কে সস্তা, কে দামি?
- **Cash flow projection** — সামনে কত টাকা ঢুকবে/বের হবে?
- **Dead stock** — কোন মাল পড়ে আছে, নড়ছেই না?

হিসাবের আসল কাজ MongoDB-র aggregation pipeline করে (ডেটাবেজের ভেতরেই) → খুব দ্রুত।

## ৫.৭ — Notification মডিউল (খবর/বিজ্ঞপ্তি 🔔)

অ্যাপের ভেতরের **নোটিফিকেশন ফিড** (ফেসবুকের ঘণ্টা আইকনের মতো)। "তুলা কম", "PO অনুমোদন দরকার",
"ডেলিভারি দেরি" — এসব আসে।

দুইটা ধারণা:
1. **Per-user** — একটা ঘটনায় যাদের জানা দরকার, প্রত্যেকের **আলাদা কপি** (fan-out at write time)
2. **90-day TTL** — ৯০ দিন পর পুরোনো নোটিফিকেশন **নিজে মুছে যায়**

কাজ: List, Unread count, Mark-read (একটা/কয়েকটা/সব)।

## ৫.৮ — Billing মডিউল (সাবস্ক্রিপশন/টাকা 💳)

সিস্টেমটাকে আসল **প্রোডাক্ট** বানায়। কোন factory কোন প্ল্যানে, কত টাকা — এসব।

মডেল: **Subscription** (প্রতি tenant একটা — AI কোটা ঠিক করে), **Invoice**, **PaymentAttempt**।

কাজ: প্ল্যান তালিকা, subscription দেখা, tier বদলানো, বাতিল, invoice তালিকা।

### ⚠️ এখনো অসম্পূর্ণ (সততার সাথে)
- **`checkout-session` → 501** = "এখনো বানানো হয়নি"। পেমেন্ট গেটওয়ে যুক্ত হয়নি।
- **Webhook (Stripe + SSLCommerz)** — এখন শুধু **লগ রাখে**, **signature যাচাই করে না** (stub)।
  প্রোডাকশনের আগে শেষ করতে হবে।

## ৬. সবচেয়ে গুরুত্বপূর্ণ — Tenant কীভাবে আলাদা থাকে? 🔐

ধরুন আপনি এই সিস্টেমে **৫টা গার্মেন্টস** চালান। তাদের ডেটা কীভাবে আলাদা থাকে?

### মূল ধারণা: একটাই ডেটাবেজ, কিন্তু প্রতিটা সারিতে একটা "tenantId" সিল

৫টা গার্মেন্টস **একই MongoDB ডেটাবেজ, একই কালেকশন** শেয়ার করে। আলাদা রাখে একটামাত্র কলাম —
**`tenantId`** — আর ৩টা নিরাপত্তা স্তর।

```
        একটাই `items` কালেকশন
   ┌────────────────────────────────────────────────────┐
   │ tenantId      | item          | stock                │
   ├────────────────────────────────────────────────────┤
   │ aaa...001 🏭  | তুলা COTTON-001 | ৬৫০   ← ঢাকা ডেনিম  │
   │ bbb...002 🏭  | তুলা COTTON-001 | ১২০০  ← চট্টগ্রাম    │
   │ ccc...003 🏭  | সুতা YARN-50    | ৩০০   ← গাজীপুর     │
   │ ddd...004 🏭  | তুলা COTTON-001 | ৮০    ← সাভার       │
   │ eee...005 🏭  | জিপার ZIP-8     | ৫০০০  ← নারায়ণগঞ্জ │
   └────────────────────────────────────────────────────┘
```

### ৩টা নিরাপত্তা স্তর

**স্তর ১ — tenantId শুধু JWT থেকে আসে**
Login করলে JWT-তে `tenantId` সিল মারা হয়। ঢাকা ডেনিম চাইলেও `bbb...002` বসাতে পারবে না (সিল
জাল করা অসম্ভব)। URL/body/header থেকে আসা কোনো tenantId **পুরোপুরি উপেক্ষা** করা হয়।

**স্তর ২ — প্রতিটা query-তে tenantId অটো যোগ হয় (tenancy plugin)**
ঢাকা ডেনিম লেখে:
```
Item.find({ type: 'raw_material' })
```
সিস্টেম চুপচাপ বদলায়:
```
Item.find({ type: 'raw_material', tenantId: aaa...001 })  ← অটো যোগ
```
ফলে চট্টগ্রামের তুলা **কখনো দেখা যাবে না**। আর scope না থাকলে query **থেমে যায়** (fail-closed) —
ভুল করেও সব tenant-এর ডেটা দেখায় না।

**স্তর ৩ — সরাসরি ID দিয়ে চুরি ঠেকানো (assertTenantOwns)**
কেউ অন্য গার্মেন্টসের PO-র ID দিয়ে দেখতে চাইলে → **404 Not Found** (403 না, ইচ্ছে করে — যাতে
"আছে কিন্তু তোমার না" এটাও ফাঁস না হয়)। চেষ্টাটা লগ হয়।

### বোনাস: একই নাম, কোনো সমস্যা নাই
৩টা গার্মেন্টসের আইটেমের SKU একই "COTTON-001" — তবু সংঘর্ষ নাই, কারণ uniqueness =
**(tenantId + sku)**, শুধু sku না। প্রতিটা index `tenantId` দিয়ে শুরু হয় → query-ও দ্রুত হয়।

### একটা বাস্তব ঘটনা (পুরো চেইন একসাথে)

> forecast করতে বলে → **Billing** চেক করে কোটা আছে কিনা → AI বলে "১০০০ গজ লাগবে" → মালিক

> (supplier-কে PDF ইমেইল) → মাল এলে **Receive** → Inventory বাড়ে, alert বন্ধ, AI আবার forecast →
> সপ্তাহ শেষে **AI রিপোর্ট** + **RPT অ্যানালিটিক্স** পুরো ছবি দেখায়।
>
> **আর এই পুরোটা ঘটে শুধু ঢাকা ডেনিম-এর (aaa...001) গণ্ডির ভেতরে** — অন্য ৪টা গার্মেন্টসের ডেটার
> সাথে কখনো মেশে না।

## ৮. এক নজরে মনে রাখার টেবিল

| মডিউল | সহজ অর্থ | মূল কথা |
|---|---|---|
| **Auth** 🚪 | দারোয়ান + পরিচয়পত্র | tenant এখানে জন্ম নেয়, JWT-তে সিল মারে |
| **Inventory** 📦 | গুদামের সত্য খাতা | append-only ledger; সত্যের উৎস |
| **Supplier** 🤝 | কার কাছ থেকে কিনব | RFQ → তুলনা → Accept করলে অটো PO |
| **PO** 📝 | আসল কেনাকাটা | কড়া state machine; Receive-এ সব মডিউল যুক্ত |
| **AI** 🧠 | ভবিষ্যৎ + রিপোর্ট | কখনো খালি হাতে ফেরে না; খরচে সীমা |
| **RPT** 📊 | অ্যানালিটিক্স | read-only, খাঁটি অংক, AI নাই |
| **Notification** 🔔 | খবর পৌঁছায় | প্রতি ইউজারের আলাদা কপি, ৯০ দিনে মুছে |
| **Billing** 💳 | সাবস্ক্রিপশন | AI কোটা ঠিক করে; পেমেন্ট অংশ অসম্পূর্ণ |

**তিন লাইনে পুরো সিস্টেম:**
1. **Auth** পরিচয় দেয় আর ৫টা গার্মেন্টসকে আলাদা রাখে।
2. **Inventory + Supplier + PO** মিলে মাল কেনা আর স্টক সামলায়।
3. **AI + RPT + Notification + Billing** মিলে বুদ্ধি, রিপোর্ট, খবর আর টাকার হিসাব দেয়।
